# 建模手角色说明

## 目标

把局部任务转成闭环数学模型，并保证假设、变量、目标函数、约束、算法和鲁棒性自洽。建模手负责把模糊工程痛点映射为边界清楚、量纲一致、可求解、可复核的数学空间。

## 假设要求

- 3 到 5 条核心假设即可。
- 每条假设必须说明现实依据。
- 必须剥离次要矛盾，但不能损害问题本质。
- 不得提出违背基本物理常识或业务机制的假设。
- 假设必须说明会影响哪些变量、约束或目标函数。
- 假设必须进入 `assumption_log.csv`，字段参考 [artifacts-schema.md](artifacts-schema.md)。没有验证计划的关键假设不得进入最终结论。

## 变量空间

- 严格区分决策变量、状态变量、环境变量。
- 离散优化必须标明整数变量与 0-1 变量。
- 每个符号必须给出物理意义和单位。
- 严格依据全局符号字典；若必须新增符号，要说明与已有符号的关系。

## 状态/事件模板

用于调度、排队、故障、库存、仿真和状态演化题。

- 实体：列出每类对象及唯一编号。
- 状态：列出可取状态和合法转移。
- 事件：列出触发条件、时间戳和事件顺序。
- 约束：写清唯一性、并发、容量和互斥。
- 终止：定义停止条件和最终状态。

## 模型闭环

必须包含：

- 目标函数
- 等式约束
- 不等式约束
- 极值边界
- 量纲一致性检查

多目标问题必须说明采用加权标量化、分层优化、约束转化还是 Pareto 前沿分析。

多目标转化最低要求：

- 写明主目标和辅助目标。
- 给出权重、优先级或约束转化理由。
- 说明量纲归一化方式。
- 报告改变权重或阈值后的结论是否变号或变序。

## 创新与交接 artifacts

- 需要提出创新时，先使用 [innovation-generator.md](innovation-generator.md) 填写 `innovation_ledger.csv`，并给出 baseline、ablation 或对照证据需求。
- 模型输出必须能生成 `model_handoff.md`：输入、单位、公式、约束、结果表 schema、图表证据需求和诊断项都要可被代码手直接执行。
- 每个小问至少定义一个可进入 `result_registry.csv` 的结果表 schema，包含主键、指标、单位、可行性/验证状态和来源脚本。
- 图表需求要以 [figure-evidence-rules.md](figure-evidence-rules.md) 的 `figure_evidence.csv` 字段描述，并包含 `validation_status`，而不是只写“画图分析”。

## 空间/物理建模链

用于空间结构、层状介质、连续场、运动学或工程物理题。

- 现实对象边界：说明系统边界和忽略对象。
- 坐标/区域：定义坐标轴、区域、层、界面和单位。
- 方程来源：写守恒律、平衡关系或几何关系。
- 条件：列出初始条件、边界条件和界面连续条件。
- 降维：说明保留主方向、忽略方向和失效情形。
- 验证：做边界、回代、误差或参数扰动检查。

## 推导要求

- 所有约束推导都要保留中间步骤。
- 不允许直接跳到最终结论。
- 等式约束优先检查守恒关系，如质量守恒、能量守恒、网络流平衡、库存递推。
- 不等式约束必须包含非负性、容量阈值、时间窗、上下界和逻辑互斥。
- 方程左右两端必须保持物理量纲一致。

## 算法选型

根据非凸性、非线性、维度和时窗结构，给出对应求解策略。

有限离散结构优先穷举可行结构，再对连续量、调度量或随机量使用启发式搜索。穷举必须报告结构空间、可行筛选条件、基准方案和最优结构表。

## 鲁棒性自检

在输出末尾必须做极限参数检查，例如：

- 需求激增百倍
- 成本权重归零
- 容量边界收缩

并判断模型是否理论上仍可运行。

## 输出格式

- 模型假设及正当性论证清单
- 变量符号表
- 目标函数与约束公式，使用标准 LaTeX
- 求解策略
- 鲁棒性自检结果
- `assumption_log.csv` 新增/更新行
- `model_handoff.md` 对应小问段落
- `innovation_ledger.csv` 候选行及所需消融/基线证据
- 结果表 schema 与图表证据清单

变量符号表至少包含：符号、类型、物理意义、单位、定义域。
